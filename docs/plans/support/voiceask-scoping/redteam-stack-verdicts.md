# RedTeam Stack Verdicts — voiceask-scoping

Adversarial validation of load-bearing claims in the three scout reports. Method: refute-by-default; every verdict rests on primary evidence from this host and the repos, not the reports. Verified 2026-07-13 on Darwin 25.5.0 / macOS 26.5.2.

| # | Claim | Verdict | Evidence |
|---|-------|---------|----------|
| 1 | whisper.cpp ships built-in Silero VAD (`--vad` / `--vad-model` flags) | **CONFIRMED** | `whisper-cli --help` (and `whisper-cpp --help`) both print a "Voice Activity Detection (VAD) options" block: `--vad`, `-vm/--vad-model FNAME`, `-vt/--vad-threshold`, `-vspd/--vad-min-speech-duration-ms`, `-vsd/--vad-min-silence-duration-ms`, `-vmsd/--vad-max-speech-duration-s`, `-vp/--vad-speech-pad-ms`, `-vo/--vad-samples-overlap`. Local build = `whisper-cpp 1.9.1` (brew, bottled, linked). Caveat: the VAD **model** `.bin` is a separate download (not bundled); the flags/engine are present in the installed binary. |
| 2 | `yap` requires macOS 26; this Mac qualifies | **CONFIRMED** | `sw_vers` → `ProductVersion: 26.5.2` (Darwin 25.5.0 maps to macOS 26 "Tahoe"), so the host qualifies. `yap` is **not installed** (`which yap` → not found). `brew info yap` resolves cleanly: `yap: stable 1.2.0 (bottled)`, uses "On-device audio transcription using Speech.framework". The macOS-26 floor is consistent with yap depending on the Speech.framework `SpeechAnalyzer` API (macOS 26), though I did not independently read yap's source to confirm the exact minimum — the practical claim (this Mac can run it, one `brew install yap` away) holds. |
| 3 | "onnxruntime-node crashes under Bun" vs "VoiceLayer runs onnxruntime-node under Bun in production" | **REFUTED (crash claim) / reconciled** | The two are not both true for VL's setup — VL works. `src/vad.ts:25` does `const ort = require("onnxruntime-node")`; the adjacent AIDEV-NOTE (`vad.ts:23-24`) states "onnxruntime-node uses native C++ addon (N-API). Works with Bun 1.3+." `package.json` pins `"onnxruntime-node": "^1.24.2"`, `"engines": { "bun": ">=1.0.0" }`, `"start": "bun run src/mcp-server.ts"`, bin → `src/mcp-server.ts`. Installed runtime is **Bun 1.3.14** (`bun --version`), satisfying the "Bun 1.3+" floor. Reconciled truth: onnxruntime-node's N-API addon runs under Bun in VL's specific, version-pinned setup (Bun ≥1.3, ORT 1.24.x) — no Node fallback in the run path. The generic "crashes under Bun" was true for older Bun/ORT combos; it does not apply here. |
| 4 | VoiceLayer's `voice_ask` has NO real-time barge-in in the shipping path (only in unwired `voicesdk/`) | **CONFIRMED** | Shipping `converse` flow (`src/handlers.ts:288-318`) is strictly sequential: `speak(..., { waitForPlayback: true })` **blocks until the question finishes**, *then* `waitForInput(...)` records — mic is never monitored during playback. The real mic-during-playback monitor `monitorDuringPlayback` is called only at `src/voicesdk/session.ts:175` (plus one test). `voicesdk` is imported by **zero** non-test production files (`grep -rln voicesdk src/*.ts` → empty). The `SoundLayer` that owns `bargeIn` (`src/soundlayer/default.ts:89`) is never constructed in the shipping path (`createDefaultSoundLayer`/`defaultSoundLayer` have no non-soundlayer callers). The one `bargeIn` in the live path (`src/tts.ts:671`, `if (priority === "critical") this.bargeIn(job)`) is **playback-queue preemption**, not user-voice barge-in. |
| 5 | sox `silence` effect can do start-on-sound + stop-after-silence endpointing | **CONFIRMED** | `sox` and `rec` installed (`/opt/homebrew/bin/sox`, `/opt/homebrew/bin/rec`). `sox --help-effect silence` → `silence [ -l ] above_periods [ duration threshold[d|%] ] [ below_periods duration threshold[d|%] ]`. `above_periods` (trim leading silence until sound = start-on-sound) + `below_periods` with duration/threshold (stop after trailing silence) provide exactly the endpointing described. |
| 6 | Echo is on macOS with mic-less daemon; `com.echo` LaunchAgent exists | **CONFIRMED** | `ls ~/Library/LaunchAgents/com.echo.plist` → `-rw-r--r--@ 1 ed staff 1081 Jul 9 19:29 .../com.echo.plist`. Host is macOS 26.5.2. (Mic-less daemon is by design per repo AGENTS.md; the plist's existence is confirmed here.) |

## Bottom line

All six claims survive adversarial checks. Five are **CONFIRMED** outright. Claim 3's apparent contradiction resolves cleanly: onnxruntime-node does **not** crash under VL's Bun setup — it runs on Bun 1.3.14 with ORT 1.24.x via N-API, exactly as the code comment asserts. The only qualification worth carrying forward: whisper.cpp's VAD needs a separately-downloaded model file (flags/engine are present, model is not bundled), and yap is one `brew install` away (not currently installed).
